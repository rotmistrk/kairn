# v-011.03 — Panel System

Parent: v-011-kairn-spec.md

## Overview

The panel system composes kairn's workspace from txv-widgets components.
It defines how `App` integrates with `EventLoop`, how panels are laid out
across three layout modes, how focus moves between panels, and how the
render/input cycles work.

## App and EventLoop composition

`App` does not implement `Widget`. It is the top-level owner that runs
inside the `EventLoop::run` closure. The event loop owns the `Screen`;
`App` owns all panel state and layout.

```rust
pub struct App {
    layout: LayoutState,
    editor_panel: EditorPanel,
    bottom_panel: BottomPanel,
    status_bar: AppStatusBar,
    focus: PanelFocus,
    mode: WindowMode,
    quit: bool,
}

impl App {
    pub fn new() -> Self;

    /// Called once per loop iteration from the EventLoop handler.
    pub fn tick(
        &mut self,
        ctx: &mut RunContext<'_>,
    ) -> LoopControl;
}
```

The `main.rs` entry point wires them together:

```rust
fn run() -> io::Result<()> {
    let screen = Screen::new(cols, rows);
    let mut event_loop = EventLoop::new(screen);
    let mut app = App::new();

    event_loop.run(|ctx| app.tick(ctx))
}
```

Inside `App::tick`:

1. Process `ctx.events` — dispatch input to focused panel.
2. Process `ctx.poll_data` — route PTY output to terminal panels.
3. Recompute layout if terminal resized.
4. Render all panels to `ctx.screen`.
5. Return `LoopControl::Quit` if `app.quit` is set.

## Panel types

### EditorPanel (triptych)

The main editing area: tree + editor + control, arranged horizontally.
Uses two `SplitPane` instances for the internal splits.

```rust
pub struct EditorPanel {
    tree: TreePanel,
    editor: EditorView,
    control: ControlPanel,
    /// tree | (editor | control)
    outer_split: SplitPane,
    /// editor | control (within the right side of outer_split)
    inner_split: SplitPane,
    tree_visible: bool,
    control_visible: bool,
    focus: TriptychFocus,
    /// Tab bar for open files (tabs mode).
    file_tabs: TabBar,
}

#[derive(Clone, Copy, PartialEq, Eq)]
pub enum TriptychFocus {
    Tree,
    Editor,
    Control,
}
```

`outer_split` is `SplitDirection::Horizontal` with divider at the tree
width (default 20, min 10, max 40). `inner_split` is
`SplitDirection::Horizontal` with divider computed from the right edge
minus control width (default 25, min 15, max 40).

When `tree_visible` is false, the tree rect collapses and the editor
takes its space. When `control_visible` is false, the control rect
collapses similarly.

### BottomPanel (tabbed)

A tabbed container for terminal, kiro, errors, search, and tests.

```rust
pub struct BottomPanel {
    tab_bar: TabBar,
    tabs: Vec<BottomTab>,
    visible: bool,
}

pub enum BottomTab {
    Terminal(TerminalPanel),
    Kiro(TerminalPanel),
    Errors(ErrorList),
    Search(SearchResults),
    Tests(TestResults),
    Output(OutputPanel),
}
```

The `TabBar` widget renders the tab strip in the first row. The active
tab's content fills the remaining rows. Tab switching is handled by the
parent (`App`) via `Ctrl-Shift-↑/↓` when the bottom panel is focused.

### AppStatusBar

Wraps the txv-widgets `StatusBar` with kairn-specific state.

```rust
pub struct AppStatusBar {
    bar: StatusBar,
    prompt: Option<InputLine>,
}
```

When `prompt` is `Some`, the status bar row is replaced by the `InputLine`
(for ex-commands `:` or search `/`). The `InputLine` receives all input
until confirmed or cancelled.

## Focus management

Focus operates at two levels: panel-level and intra-panel.

### Panel-level focus

```rust
#[derive(Clone, Copy, PartialEq, Eq)]
pub enum PanelFocus {
    Editor,
    Bottom,
    Prompt,
}
```

`Ctrl-X ↓` moves focus from editor to bottom. `Ctrl-X ↑` moves back.
When a prompt is active, `PanelFocus::Prompt` overrides — all input goes
to the `InputLine` until it resolves.

### Intra-panel focus

Within `EditorPanel`, `TriptychFocus` tracks which sub-panel has focus.
`Ctrl-X ←/→` cycles between tree, editor, and control. The `FocusGroup`
widget from txv-widgets is not used at the panel level because panels are
not homogeneous `Box<dyn Widget>` — they have distinct types and distinct
rendering surfaces. Focus cycling is handled by `App` directly.

Within `BottomPanel`, the active tab's widget receives all input.

### Focus dispatch rules

1. If `focus == Prompt`, all keys go to `status_bar.prompt`.
2. If `focus == Bottom`, keys go to `bottom_panel.active_tab()`.
3. If `focus == Editor`, keys go to `editor_panel` which routes to
   its `TriptychFocus` target.
4. Global keys (`Ctrl-Q`, `Ctrl-L`, `Ctrl-B`, `Ctrl-X` sequences) are
   checked first by `App` before dispatching to panels.

## Layout computation

### LayoutMode

```rust
    TallBottom,
}
```

### LayoutState

```rust
pub struct LayoutState {
    mode: LayoutMode,
    screen_w: u16,
    screen_h: u16,
    tree_width: u16,
    control_width: u16,
    bottom_height_pct: u16,
}
```

### Layout-to-Rect mapping

All layout computation uses `Rect::split` from `txv::layout`. The screen
is split in two passes: a vertical split (top area vs bottom + status),
then a horizontal split within the top area.

```
Full screen rect
├── Vertical split
│   ├── top_area (Fill)
│   ├── bottom_area (Percent or Fixed, if visible)
│   └── status_row (Fixed 1)
```

Then `top_area` is split horizontally per mode:

**Wide** (tree | editor | control):

```rust
let cols = top_area.split(
    Direction::Horizontal,
    &[
        Constraint { size: Size::Fixed(tree_width), min: 10, max: 40 },
        Constraint { size: Size::Fill, min: 20, max: u16::MAX },
        Constraint { size: Size::Fixed(control_width), min: 15, max: 40 },
    ],
);
// cols[0] = tree, cols[1] = editor, cols[2] = control
```

**TallRight** (tree | editor+bottom stacked):

```rust
let cols = top_area.split(
    Direction::Horizontal,
    &[
        Constraint { size: Size::Fixed(tree_width), min: 10, max: 40 },
        Constraint { size: Size::Fill, min: 20, max: u16::MAX },
    ],
);
// cols[0] = tree (full height)
// cols[1] is then split vertically:
let rows = cols[1].split(
    Direction::Vertical,
    &[
        Constraint { size: Size::Fill, min: 5, max: u16::MAX },
        Constraint { size: Size::Percent(bottom_pct), min: 3, max: ... },
    ],
);
// rows[0] = editor, rows[1] = bottom
```

**TallBottom** (tree+editor side-by-side, bottom full width):

```rust
// First vertical split: upper area vs bottom
let rows = screen_area.split(
    Direction::Vertical,
    &[
        Constraint { size: Size::Fill, min: 5, max: u16::MAX },
        Constraint { size: Size::Percent(bottom_pct), min: 3, max: ... },
        Constraint { size: Size::Fixed(1), min: 1, max: 1 },
    ],
);
// rows[0] = upper, rows[1] = bottom, rows[2] = status

let cols = rows[0].split(
    Direction::Horizontal,
    &[
        Constraint { size: Size::Fixed(tree_width), min: 10, max: 40 },
        Constraint { size: Size::Fill, min: 20, max: u16::MAX },
    ],
);
// cols[0] = tree, cols[1] = editor
```

### Auto-select on resize

When the terminal is resized, `LayoutState` auto-selects the mode based
on width thresholds (>160 → Wide, 100–160 → TallRight, <100 →
TallBottom) unless the user has manually cycled with `Ctrl-L`, in which
case the manual choice is preserved until the next resize crosses a
threshold boundary.

```rust
impl LayoutState {
    pub fn auto_mode(width: u16) -> LayoutMode;

    pub fn compute_rects(&self) -> PanelRects;
}

pub struct PanelRects {
    pub tree: Option<Rect>,
    pub editor: Rect,
    pub control: Option<Rect>,
    pub bottom: Option<Rect>,
    pub status: Rect,
    pub file_tabs: Option<Rect>,
}
```

`tree` is `None` when hidden. `control` is `None` when hidden or in
TallRight/TallBottom mode. `bottom` is `None` when hidden. `file_tabs`
is `Some` (1-row rect above the editor area) when in tabs mode with
multiple open files.

## Panel visibility toggles and resize

| Key | Action | Field |
|-----|--------|-------|
| `Ctrl-B` | Toggle tree panel | `editor_panel.tree_visible` |
| `Ctrl-X P` | Toggle control panel | `editor_panel.control_visible` |
| `Ctrl-X \` | Toggle bottom panel | `bottom_panel.visible` |
| `Ctrl-L` | Cycle layout mode | `layout.mode` |
| `F7` / `F8` | Shrink / grow tree width by 1 | `layout.tree_width` |
| `Shift-F7` / `Shift-F8` | Shrink / grow tree width by 5 | `layout.tree_width` |
| `F9` / `F10` | Shrink / grow bottom height by 1 row | `layout.bottom_height_pct` |

When a panel is toggled off, its rect is reclaimed by the editor. Focus
moves to the editor if the hidden panel had focus.

Resize is clamped by `Constraint::min` / `Constraint::max` in the layout
split. The `LayoutState` stores absolute cell values for tree/control
widths and a percentage for bottom height.

## Windows vs tabs mode

```rust
#[derive(Clone, Copy, PartialEq, Eq)]
pub enum WindowMode {
    /// One file visible at a time, tab bar shows all open files.
    Tabs,
    /// Screen splits to show multiple editor triptychs.
    Windows,
}
```

### Tabs mode (default)

A single `EditorPanel` is active. The `TabBar` at the top of the editor
area shows all open files. `Alt-Left` / `Alt-Right` switches tabs.
`Ctrl-W` closes the current tab.

### Windows mode

Multiple `EditorPanel` instances arranged via `SplitPane`. Each window
has its own tree+editor+control. The bottom panel and status bar remain
shared.

```rust
pub struct WindowManager {
    windows: Vec<EditorPanel>,
    splits: Vec<SplitPane>,
    focused_window: usize,
}
```

`Ctrl-X W` toggles between modes. In windows mode, `Ctrl-X 1` closes
all windows except the focused one. `Ctrl-X 2` splits horizontally.
`Ctrl-X 3` splits vertically. `Ctrl-X O` cycles focus between windows.

Windows mode is a Phase D feature. The initial implementation supports
tabs mode only.

## Render cycle

Each `App::tick` call ends with a render pass:

```rust
impl App {
    fn render(&self, screen: &mut Screen) {
        let rects = self.layout.compute_rects();

        // 1. File tab bar (if visible)
        if let Some(tab_rect) = rects.file_tabs {
            let mut s = screen.surface(
                tab_rect.x, tab_rect.y,
                tab_rect.w, tab_rect.h,
            );
            self.editor_panel.file_tabs.render(
                &mut s,
                false,
            );
        }

        // 2. Tree panel (if visible)
        if let Some(tree_rect) = rects.tree {
            let mut s = screen.surface(
                tree_rect.x, tree_rect.y,
                tree_rect.w, tree_rect.h,
            );
            let focused = self.focus == PanelFocus::Editor
                && self.editor_panel.focus == TriptychFocus::Tree;
            self.editor_panel.tree.render(&mut s, focused);
        }

        // 3. Editor
        {
            let r = rects.editor;
            let mut s = screen.surface(r.x, r.y, r.w, r.h);
            let focused = self.focus == PanelFocus::Editor
                && self.editor_panel.focus == TriptychFocus::Editor;
            self.editor_panel.editor.render(&mut s, focused);
        }

        // 4. Control panel (if visible)
        if let Some(ctrl_rect) = rects.control {
            let mut s = screen.surface(
                ctrl_rect.x, ctrl_rect.y,
                ctrl_rect.w, ctrl_rect.h,
            );
            let focused = self.focus == PanelFocus::Editor
                && self.editor_panel.focus == TriptychFocus::Control;
            self.editor_panel.control.render(&mut s, focused);
        }

        // 5. Bottom panel (if visible)
        if let Some(bot_rect) = rects.bottom {
            let mut s = screen.surface(
                bot_rect.x, bot_rect.y,
                bot_rect.w, bot_rect.h,
            );
            self.bottom_panel.render(
                &mut s,
                self.focus == PanelFocus::Bottom,
            );
        }

        // 6. Status bar (always)
        {
            let r = rects.status;
            let mut s = screen.surface(r.x, r.y, r.w, r.h);
            self.status_bar.render(
                &mut s,
                self.focus == PanelFocus::Prompt,
            );
        }
    }
}
```

Each panel receives a `Surface` scoped to its computed `Rect`. Panels
render independently — no panel can write outside its surface bounds
(enforced by `Surface` clipping).

## Input cycle

```rust
impl App {
    fn handle_events(&mut self, events: &[Event]) {
        for event in events {
            match event {
                Event::Key(key) => self.handle_key(*key),
                Event::Resize(w, h) => self.handle_resize(*w, *h),
                _ => {}
            }
        }
    }

    fn handle_key(&mut self, key: KeyEvent) {
        // 1. Global keys (checked first)
        if self.handle_global_key(key) {
            return;
        }

        // 2. Dispatch to focused panel
        let result = match self.focus {
            PanelFocus::Prompt => {
                self.status_bar.handle_key(key)
            }
            PanelFocus::Bottom => {
                self.bottom_panel.handle_key(key)
            }
            PanelFocus::Editor => {
                self.editor_panel.handle_key(key)
            }
        };

        // 3. Handle actions bubbled up from panels
        match result {
            EventResult::Action(action) => {
                self.handle_action(action);
            }
            _ => {}
        }
    }

    /// Returns true if the key was consumed as a global binding.
    fn handle_global_key(&mut self, key: KeyEvent) -> bool;

    fn handle_action(&mut self, action: WidgetAction);

    fn handle_resize(&mut self, w: u16, h: u16);
}
```

Global keys are processed before panel dispatch. If a global key matches,
the panel never sees it. Global keys include:

- `Ctrl-Q` — quit
- `Ctrl-L` — cycle layout
- `Ctrl-B` — toggle tree
- `Ctrl-X` prefix — starts a two-key chord sequence
- `F7`/`F8`/`F9`/`F10` — resize panels

The two-key chord (`Ctrl-X` followed by another key) is tracked in `App`
state:

```rust
pub struct App {
    // ... other fields ...
    pending_chord: Option<KeyEvent>,
}
```

When `pending_chord` is `Some`, the next key completes the chord and is
handled by `App` (e.g., `Ctrl-X ↓` = focus bottom, `Ctrl-X P` = toggle
control).

## Panel trait

Panels share a common interface but are not `dyn Widget` because they
need typed access for panel-specific operations (e.g., `EditorPanel`
exposes `open_file`, `BottomPanel` exposes `switch_tab`). Instead, each
panel implements the `Widget` trait from txv-widgets directly:

```rust
impl Widget for EditorPanel {
    fn render(&self, surface: &mut Surface<'_>, focused: bool);
    fn handle_key(&mut self, key: KeyEvent) -> EventResult;
}

impl Widget for BottomPanel {
    fn render(&self, surface: &mut Surface<'_>, focused: bool);
    fn handle_key(&mut self, key: KeyEvent) -> EventResult;
}

impl Widget for AppStatusBar {
    fn render(&self, surface: &mut Surface<'_>, focused: bool);
    fn handle_key(&mut self, key: KeyEvent) -> EventResult;
}
```

`App` calls these through concrete types, not trait objects.
