# v-011.05 — LSP Client

Parent: [v-011-kairn-spec.md](v-011-kairn-spec.md)
Depends on: [v-011.01](v-011.01-piece-table.md) (PieceTable.take_changes for incremental sync)

## Overview

kairn embeds an LSP client that communicates with language servers over
stdio using JSON-RPC 2.0. One server process per language. Servers start
lazily on first file open and shut down when the last file of that
language closes. The client runs on tokio tasks; all UI interaction
happens on the main thread via channels.

## Architecture

```
┌──────────────────────────────────────────────────────┐
│                    Main thread                       │
│                                                      │
│  EditorPanel ──→ LspClient ──→ mpsc::Sender<LspReq> │
│       ↑                                              │
│       └──── mpsc::Receiver<LspEvent> ←───────────┐   │
└──────────────────────────────────────────────────┐│───┘
                                                   ││
┌──────────────────────────────────────────────────┘│───┐
│                  Tokio runtime                    │   │
│                                                   ↓   │
│  ┌─────────────┐   stdio    ┌──────────────────┐     │
│  │ writer task  │ ────────→ │  Language server  │     │
│  └─────────────┘            │  (gopls, jdtls,  │     │
│  ┌─────────────┐   stdio    │   rust-analyzer,  │     │
│  │ reader task  │ ←──────── │   tsserver, etc.) │     │
│  └─────────────┘            └──────────────────┘     │
└──────────────────────────────────────────────────────┘
```

- **LspClient** lives on the main thread. It exposes synchronous methods
  that send requests through a channel and return immediately.
- **Writer task** reads from the request channel, serializes JSON-RPC
  messages, and writes them to the server's stdin.
- **Reader task** reads from the server's stdout, parses JSON-RPC
  messages, and sends events back to the main thread.
- **One LspServer** per language — holds the child process handle, the
  channel endpoints, and the set of open document URIs.

## Server lifecycle

### Lazy start

Servers are not started at application launch. When a file is opened:

1. LspClient looks up the file extension in the registry.
2. If no ServerConfig matches, the file gets no LSP support (silent).
3. If a config matches but the server is not running, LspClient spawns it.
4. The spawn sequence:
   a. Start child process via `tokio::process::Command` with stdio piped.
   b. Send `initialize` request with workspace root and client capabilities.
   c. Wait for `initialize` response.
   d. Send `initialized` notification.
   e. Send `textDocument/didOpen` for the triggering file.
5. If the server binary is not found, log a warning and mark the language
   as unavailable for this session. Do not retry.

### Shutdown

When the last file of a language closes:

1. Send `textDocument/didClose` for the file.
2. Check if any other documents remain open for this server.
3. If none remain:
   a. Send `shutdown` request.
   b. Wait up to 5 seconds for response.
   c. Send `exit` notification.
   d. If the process does not exit within 2 seconds, kill it.
4. Remove the server from the active servers map.

### Crash recovery

If the reader task detects EOF on stdout (server crashed):

1. Log the crash with the server's stderr output.
2. Notify the main thread via `LspEvent::ServerCrashed`.
3. Mark all documents from that server as "no LSP".
4. On the next edit or file open for that language, attempt one restart.
5. If the restart also crashes within 30 seconds, disable that language
   server for the session.

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum ServerState {
    /// Not started yet.
    Idle,
    /// Initialize request sent, waiting for response.
    Starting,
    /// Running and ready for requests.
    Ready,
    /// Shutdown request sent, waiting for response.
    ShuttingDown,
    /// Crashed or exited unexpectedly.
    Crashed,
    /// Permanently disabled after repeated crashes.
    Disabled,
}
```

## JSON-RPC protocol

### Wire format

Messages use the LSP base protocol: HTTP-style headers followed by a
JSON body.

```
Content-Length: 73\r\n
\r\n
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}
```

### Message types

```rust
/// Outgoing message from client to server.
#[derive(Debug, Clone)]
pub enum LspMessage {
    Request(LspRequest),
    Notification(LspNotification),
}

/// A request expects a response. Identified by a numeric ID.
#[derive(Debug, Clone)]
pub struct LspRequest {
    pub id: RequestId,
    pub method: String,
    pub params: serde_json::Value,
}

/// A notification is fire-and-forget. No response expected.
#[derive(Debug, Clone)]
pub struct LspNotification {
    pub method: String,
    pub params: serde_json::Value,
}

/// Incoming message from server to client.
#[derive(Debug, Clone)]
pub enum LspIncoming {
    Response(LspResponse),
    Notification(LspNotification),
    Request(LspRequest),
}

/// Response to a client request.
#[derive(Debug, Clone)]
pub struct LspResponse {
    pub id: RequestId,
    pub result: Option<serde_json::Value>,
    pub error: Option<LspError>,
}

#[derive(Debug, Clone)]
pub struct LspError {
    pub code: i32,
    pub message: String,
    pub data: Option<serde_json::Value>,
}

/// Request ID — always numeric in kairn's client.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct RequestId(u64);
```

### Framing

```rust
/// Reads one LSP message from a byte stream.
/// Parses Content-Length header, reads that many bytes, deserializes JSON.
fn read_message(reader: &mut impl BufRead) -> Result<LspIncoming>;

/// Writes one LSP message to a byte stream.
/// Serializes JSON, prepends Content-Length header.
fn write_message(writer: &mut impl Write, msg: &LspMessage) -> Result<()>;
```

## Document sync

kairn uses **incremental sync** (TextDocumentSyncKind::Incremental = 2).
The PieceTable's `take_changes()` method produces exactly the change
events LSP expects.

### didOpen

Sent when a file is first opened in the editor and the server is ready.

```rust
/// textDocument/didOpen parameters.
fn did_open_params(
    uri: &DocumentUri,
    language_id: &LanguageId,
    version: DocumentVersion,
    text: &str,
) -> serde_json::Value;
```

The full file content is sent. `version` starts at 0.

### didChange

Sent after each edit (debounced — see below). Uses incremental changes
from `PieceTable::take_changes()`.

```rust
/// textDocument/didChange parameters.
fn did_change_params(
    uri: &DocumentUri,
    version: DocumentVersion,
    changes: &[TextChange],
) -> serde_json::Value;
```

Each `TextChange` maps to an LSP `TextDocumentContentChangeEvent`:

```json
{
  "range": {
    "start": { "line": 0, "character": 5 },
    "end": { "line": 0, "character": 5 }
  },
  "text": " beautiful"
}
```

**Debouncing**: changes are batched for 50ms after the last keystroke.
If the user types "hello" quickly, one didChange with all accumulated
changes is sent rather than five. The debounce timer resets on each edit.

```rust
/// Semantic type for document URIs.
#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub struct DocumentUri(String);

/// Semantic type for document versions.
#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord)]
pub struct DocumentVersion(i32);
```

### didSave

Sent when the user saves the file. Includes the full text if the server
requested it in capabilities.

### didClose

Sent when the buffer is closed. The server may release resources for
this document.

### Version tracking

Each document has a monotonically increasing version number. It
increments on every didChange sent (not on every keystroke — on every
debounced batch). The version is used by the server to correlate
diagnostics with document state.

## Async architecture

### Tokio runtime

kairn runs a single tokio runtime created at startup. The main thread
is **not** a tokio task — it runs the txv-widgets EventLoop. The tokio
runtime runs on a background thread pool.

```rust
// In main.rs startup:
let rt = tokio::runtime::Builder::new_multi_thread()
    .worker_threads(2)
    .enable_all()
    .build()?;
```

### Per-server tasks

When a server starts, two tasks are spawned:

**Writer task**: reads `LspMessage` from a channel, serializes, writes
to the child's stdin.

```rust
async fn writer_task(
    mut rx: mpsc::Receiver<LspMessage>,
    mut stdin: tokio::process::ChildStdin,
) {
    while let Some(msg) = rx.recv().await {
        if let Err(e) = write_message(&mut stdin, &msg).await {
            tracing::error!("LSP write error: {e}");
            break;
        }
    }
}
```

**Reader task**: reads from the child's stdout, parses JSON-RPC, sends
`LspEvent` to the main thread.

```rust
async fn reader_task(
    language_id: LanguageId,
    mut stdout: tokio::io::BufReader<tokio::process::ChildStdout>,
    event_tx: mpsc::Sender<LspEvent>,
) {
    loop {
        match read_message(&mut stdout).await {
            Ok(msg) => {
                let event = decode_incoming(language_id.clone(), msg);
                if event_tx.send(event).await.is_err() {
                    break; // main thread dropped receiver
                }
            }
            Err(_) => {
                // EOF or parse error — server crashed
                let _ = event_tx.send(
                    LspEvent::ServerCrashed(
                        language_id.clone(),
                        "connection lost".into(),
                    )
                ).await;
                break;
            }
        }
    }
}
```

### Channel sizing

- Request channel (main → writer): bounded(64). Backpressure is
  unlikely since requests are infrequent relative to processing speed.
- Event channel (reader → main): bounded(256). Diagnostics can be
  bursty after a save.

### Polling from EventLoop

The main thread's EventLoop polls `LspClient::poll_events()` on each
tick (every 50ms). This drains the event channel non-blockingly via
`try_recv()`. Events are dispatched to the appropriate panel.

## UI integration

### Completion popup

Triggered by typing or explicit invocation (Ctrl-Space / Ctrl-N).

1. EditorPanel calls `lsp_client.request_completion(uri, line, col)`.
2. On next poll, `LspEvent::Completions` arrives.
3. EditorPanel creates an `Overlay<ListView<CompletionData>>` anchored
   below the cursor position.
4. The ListView shows completion items with kind icons and labels.
5. Typing narrows the list (client-side filtering by prefix).
6. Enter accepts the selected item — inserts `insert_text` into the buffer.
7. Esc dismisses the popup.

```
┌──────────────────────────────┐
│ fn main() {                  │
│     let x = foo.ba|          │
│              ┌──────────┐    │
│              │ ƒ bar()  │    │
│              │ ■ baz    │    │
│              │ ƒ batch()│    │
│              └──────────┘    │
│ }                            │
└──────────────────────────────┘
```

### Hover tooltip

Triggered by cursor resting on a symbol for 500ms (configurable).

1. EditorPanel starts a timer on cursor movement.
2. If the cursor stays still for 500ms, request hover.
3. On response, show an `Overlay` with the hover content as plain text.
4. Any cursor movement dismisses the tooltip.

### Diagnostics in gutter

The gutter column (left of line numbers) shows diagnostic markers:

| Severity | Gutter | Color |
|----------|--------|-------|
| Error | `E` | Red |
| Warning | `W` | Yellow |
| Info | `I` | Blue |
| Hint | `H` | Cyan |

Lines with diagnostics are also underlined in the editor with the
corresponding color.

### Error list in bottom panel

The "Errors" tab in the bottom panel shows all diagnostics across open
files, sorted by severity then file path.

```
┌─ Errors ─────────────────────────────────────────────┐
│ E src/main.rs:12:5  expected `usize`, found `&str`   │
│ E src/main.rs:24:1  missing lifetime specifier        │
│ W src/lib.rs:8:9    unused variable `x`               │
└──────────────────────────────────────────────────────┘
```

Enter on a diagnostic jumps to the file and line. The list updates
live as `LspEvent::Diagnostics` events arrive.

### Definition / references

- **Go to definition**: opens the target file and jumps to the line.
  If the file is already open, switches to that buffer.
- **Find references**: populates a list in the bottom panel "References"
  tab. Each entry shows file:line with a context snippet. Enter jumps
  to the location.

### Document symbols

The control panel's "Outline" mode is populated from
`textDocument/documentSymbol` responses. Symbols are displayed as a
TreeView with nesting (e.g., methods inside classes). Selecting a
symbol jumps the editor cursor to that line.

### Formatting

Format command applies `TextEditRange` edits to the buffer in reverse
order (bottom-to-top) to preserve offsets. The buffer is marked modified
if any edits were applied.

### Rename

1. `prepareRename` validates the position is renameable.
2. A prompt (InputLine in status bar) asks for the new name, pre-filled
   with the current symbol name.
3. `rename` request returns workspace edits.
4. Edits are applied to all affected open buffers. Files not currently
   open are opened, edited, and saved.

### Mock server

```rust
/// A mock LSP server for testing. Runs as an async task.
/// Reads JSON-RPC from stdin, responds with pre-configured results.
struct MockLspServer {
    responses: HashMap<String, serde_json::Value>,
}

impl MockLspServer {
    fn new() -> Self;

    /// Register a canned response for a method.
    fn on_request(
        &mut self,
        method: &str,
        response: serde_json::Value,
    );

    /// Register a notification handler that records calls.
    fn on_notification(
        &mut self,
        method: &str,
    ) -> Arc<Mutex<Vec<serde_json::Value>>>;

    /// Run the mock server on the given stdio streams.
    async fn run(
        self,
        stdin: tokio::io::BufReader<impl AsyncRead + Unpin>,
        stdout: impl AsyncWrite + Unpin,
    );
}
```

The mock server is started as a child process via a test helper that
builds a small binary, or as an in-process task using `tokio::io::duplex`
for piped streams.
